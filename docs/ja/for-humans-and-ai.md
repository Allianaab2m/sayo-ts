# 人と AI のためのフレームワーク

sayo-ts の設計思想のひとつは、「**コードを書くのが人間であれ AI であれ、同じ土俵で検査できる**」ことです。
コーディングの現場が人間単独からペアに — そして AI エージェントを含む 3 者以上の協業に — 移り変わっていく中で、フレームワーク側が用意すべきは「誰が書いたかに依存しない、共通の正しさ基準」だと考えています。

この章では、sayo-ts が採用している個々の仕組み ([03. Layer システムと DI](./03-layer-and-di.md) / [06. エラーハンドリング](./06-error-handling.md) / [10. 規約と Lint ルール](./10-conventions.md) など) が、**なぜ人と AI の両方にとって有益なのか** を簡単にまとめます。

前提章: [01. はじめに](./01-getting-started.md) を流し読みしていると読みやすいです。

---

## 背景: 共通の敵は「暗黙の規約」

フレームワークや社内コードベースに必ず存在する「チームメンバーは知っているがコードには書かれていない規約」は、**人間にとっても AI にとっても最大のつまずき** です。

- 新人エンジニアは最初の数週間でそれを学び直す
- AI コーディング補助は、その暗黙知にアクセスできないので**しばしば尤もらしい嘘 (hallucination) を書く**
- レビュワーは毎回同じ指摘を繰り返す

sayo-ts の基本姿勢は、「**暗黙の規約をやめて、規約は型と ESLint として機械可読にする**」です。これは人・AI どちらに向けても同じ価値を持ちます。

---

## 仕組み 1: 型が最初のレビュワー

Effect の `Effect.Effect<A, E, R>` は **成功・失敗・依存を 3 つとも型に載せる** 設計です。sayo-ts はこれを素直に受け入れ、次のような逸脱を **ビルド時点で拒否** します。

- `Layer.provide(UserServiceLive)` を忘れる → ビルドエラー (`R` に `UserService` が残る)
- エンドポイント `error` に宣言していないエラーをハンドラが投げる → ビルドエラー (ハンドラの `E` と `api.ts` の `error` が合わない)
- 実装していないエンドポイントがある → ビルドエラー (`HttpApiBuilder.group` が不完全)
- `Effect.fail("string")` など非 Tagged なエラー → 警告 (`@sayo-ts/tagged-error-required`)

これは人の「うっかり」にも、AI の「文脈不足による捏造」にも等しく効きます。型検査はいついかなる時も平等な検査を行います。

関連章: [02. Effect の基礎](./02-effect-essentials.md), [03. Layer システムと DI](./03-layer-and-di.md)

## 仕組み 2: 規約は ESLint で機械可読

`@sayo-ts/eslint-plugin` の 7 ルールは、いずれも「レビューで繰り返し指摘されるパターン」をそのまま機械可読にしたものです。

- `no-raw-promise` / `no-try-catch` / `no-run-sync-in-handler` — Fiber ランタイムの外に脱出させない
- `tagged-error-required` — エラー設計を型レベルに維持する
- `endpoint-response-schema-required` / `endpoint-error-schema-required` — API 契約を OpenAPI と一致させる
- `service-interface-separation` — テスト容易性を担保するディレクトリ規約

この結果:

- **人間**: レビューで毎回同じ指摘をしなくて済む
- **AI**: プロンプトに書ききれない規約を `pnpm lint` が**事後に物理強制**してくれる (= プロンプトの短さが安全性を損なわない)

どちらの側でも「生成されたコードが規約に沿っているか」を **人が頭で確認するコストが 0 に近づきます**。

関連章: [10. 規約と Lint ルール](./10-conventions.md)

## 仕組み 3: Schema は 1 箇所で完結する

`Schema` ([07. バリデーションとスキーマ](./07-validation.md)) は、**1 箇所書けば TypeScript 型・実行時バリデータ・OpenAPI スキーマ・クライアント型が同時に決まる** DSL です。

AI コーディング補助が苦手とするタスクのひとつは、「**同じ契約を複数箇所に書き分けて辻褄を合わせる**」ことです。DTO クラス・バリデーションデコレータ・OpenAPI アノテーション・TS 型・API クライアントを同時に揃えさせようとすると、どこかで食い違いが起きがちです。

sayo-ts では:

```ts
export class UserResponse extends Schema.Class<UserResponse>("UserResponse")({
  id: Schema.String,
  name: Schema.String,
  email: Schema.String,
}) {}
```

これ 1 つで上記 4 つがすべて決まります。**食い違いの発生源そのものを削除している** ので、人間も AI も間違えようがありません。

関連章: [07. バリデーションとスキーマ](./07-validation.md)

## 仕組み 4: CLI が決定的な雛形を提供する

`sayo generate <name>` ([09. CLI とスキャフォールディング](./09-cli-and-scaffolding.md)) は、**毎回同じ 6 ファイル構成・同じ命名規則** で雛形を生成します。

- **人間**: 記憶に頼らず正しい雛形から書き始められる
- **AI**: "それっぽい" 架空のディレクトリ構成を発明する必要がない。`sayo generate` の出力に続ける形で作業することで、**ファイル配置や命名の迷いがなくなる**

「まずは `npx sayo generate X` を実行してから編集する」という単純な手順を 1 つ挟むだけで、AI が生成するコードの**位置と名前**が揃うようになります。これは `AGENT.md` やハーネスで吸収するより堅い方法です。

関連章: [09. CLI とスキャフォールディング](./09-cli-and-scaffolding.md)

## 仕組み 5: ポート/アダプタの分離で差し替えが簡単

`service.ts` (ポート) と `service.live.ts` (アダプタ) を分ける規約 ([03. Layer システムと DI](./03-layer-and-di.md)) は、**テスト時の差し替えコストを最小化** します。

- **人間**: モック実装を作るために本番コードを触らなくて済む
- **AI**: 「このサービスだけを差し替えた新しい Layer を書いて」という**粒度の細かい依頼** が自然に通る。本番実装とテスト実装が明確に分かれているので、AI が本番コードをうっかり壊す確率が下がる

テスト Layer が `Layer.Layer<UserService>` という **本番と同じ型** である以上、モックがインターフェースを満たしていなければコンパイルが通りません。これも型による共通基盤の一例です。

関連章: [03. Layer システムと DI](./03-layer-and-di.md), [08. テスト](./08-testing.md)

## AI と協業するときの作法

sayo-ts は特定の AI ツールを前提としません。ただし、上記の仕組みを踏まえると、AI コーディング補助と作業する際に次の流れが推奨されます。

1. **先に `sayo generate` でリソースの雛形を作ってから AI に渡す**
   - ファイル配置・命名・インポートパスが既に正しく揃っているので、AI は実装だけに集中できる
2. **`src/api.ts` とリソース内の `api.ts` を先に埋めて、契約を型として固定する**
   - `success` / `error` / `params` / `payload` を先に確定させれば、ハンドラと service.ts の型は契約から逆算できる
3. **ハンドラ実装は Effect.gen の雛形に yield\* だけ足す形で書かせる**
   - `Effect.runSync` / `try-catch` / 生 Promise が混入しても `pnpm lint` が弾く
4. **仕上げに `pnpm lint && pnpm tsc --noEmit && pnpm test` を通す**
   - この 3 つを通過した段階で、**人が書いても AI が書いても最低限の正しさは揃っている**

逆に言えば、AI がこれら 3 つを通過できないコードを出してきた場合、それは **AI の文脈不足ではなく構造的な問題** を示しています。ルールが指摘するのはいつも同じこと — 「規約からの逸脱」です。

## 人・AI 共通のチェックリスト

PR 作成時・マージ時に次を満たしていれば、書き手によらず最低限の品質は揃っています。

- [ ] `pnpm tsc --noEmit` が通る (型エラーなし)
- [ ] `pnpm lint` が通る (`@sayo-ts/eslint-plugin` のルール違反なし)
- [ ] `pnpm test` が通る (`NodeHttpServer.layerTest` 経由の統合テスト含む)
- [ ] 新規リソースなら `src/api.ts` と `src/main.ts` への配線が入っている
- [ ] 新規エラー型は `Schema.TaggedErrorClass` で定義されている
- [ ] 新規エンドポイントは `success` と `error` を宣言している
- [ ] サービスインターフェース (`service.ts`) と実装 (`service.live.ts`) が別ファイルになっている

このリストは、**人に対してもコードエージェントに対してもそのまま渡せる** 形に設計しています。`.github/pull_request_template.md` や `AGENTS.md` にコピーして使って構いません。

## まとめ

sayo-ts は「AI 対応フレームワーク」ではなく、「**規約が型と lint として機械可読になっているフレームワーク**」です。結果として:

- 人間は「規約を口伝で教える・レビューで指摘する」コストが下がる
- AI は「プロンプトでは伝えきれない規約」を事後検査として物理強制できる

**両者が同じ正しさ基準で審判される** ことが、人と AI が一緒にコードを書く時代のフレームワークに最も必要なものだと私は考えています。
